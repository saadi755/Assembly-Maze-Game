
# 🕹️ DOS Maze Game

> A terminal-based maze navigation game written in **x86 16-bit Assembly**, running natively on DOS / DOSBox via direct video memory manipulation.

---

## Overview

This project is a **16-bit x86 Assembly** game targeting the DOS `.COM` format. The player character (`*`) spawns near the bottom of the screen and must navigate to the goal tile (`F`) near the top while avoiding wall tiles drawn with a blue attribute.

All rendering is done by **writing directly to the CGA/VGA text-mode video buffer** at segment `0xB800`, bypassing any BIOS video calls for maximum speed and control. Input is handled via **BIOS INT 16h**, and the program exits cleanly via **INT 21h** — fully compatible with real DOS and modern emulators like DOSBox.

This was developed as a systems programming assignment to demonstrate:

- Direct memory-mapped I/O (video buffer at `0xB800`)
- BIOS interrupt-driven keyboard handling
- Efficient "look-ahead" collision detection
- Branching, looping, and structured control flow in raw assembly
- Clean DOS program termination

## Features

| Feature | Description |
|---|---|
| **Direct Video Memory** | Writes tiles straight to `0xB800` — no BIOS video calls |
| **Color Attributes** | Distinct colors per tile type (blue walls, yellow player, red goal) |
| **Collision Detection** | Reads the next tile's attribute byte before committing a move |
| **Goal Detection** | Checks for the finish-line offset after every move |
| **Win Screen** | Floods the screen green on reaching the goal |
| **Clean Exit** | Uses `INT 21h / 4Ch` to return to DOS shell properly |
| **.COM Format** | Single flat binary, no relocations, starts at `0x0100` — pure simplicity |

## How It Works

### Memory Map & Video Segment

In DOS text mode (80×25), the video buffer lives at physical address `0xB8000`. Each screen cell occupies **2 bytes**:

```
Byte 0 (low)  → ASCII character code
Byte 1 (high) → Attribute byte (background color in high nibble, foreground in low nibble)
```

The segment is set up once at the start of `game_loop`:

```asm
mov ax, 0xb800
mov es, ax
```

All tile reads and writes then use `ES:DI` addressing.

### Tile Encoding

Each tile is stored as a 16-bit word in the data section:

| Constant | Value | Meaning |
|---|---|---|
| `wallChar` | `0x1E23` | Attr `0x1E` (blue bg, yellow fg) + `#` (0x23) |
| `playerChar` | `0x0E2A` | Attr `0x0E` (black bg, yellow fg) + `*` (0x2A) |
| `goalChar` | `0x4746` | Attr `0x47` (red bg, white fg) + `F` (0x46) |
| `emptyChar` | `0x0720` | Attr `0x07` (black bg, grey fg) + ` ` (0x20) |

The **attribute byte** is the high byte (`AH` after a word load), which makes collision detection trivial: just compare `AH` against `0x1E` to detect walls.

### Player Movement

The player's position is tracked as a **byte offset into the video buffer** stored in `playerPos`. Each screen row is 80 cells × 2 bytes = **160 bytes wide**:

| Direction | Offset Change |
|---|---|
| Up | `− 160` |
| Down | `+ 160` |
| Left | `− 2` |
| Right | `+ 2` |

```asm
pre_up:    sub dx, 160 | jmp check_collision
pre_down:  add dx, 160 | jmp check_collision
pre_left:  sub dx, 2   | jmp check_collision
pre_right: add dx, 2   | jmp check_collision
```

The new position is computed into `DX` **before** the move is committed, so collision detection can abort it.

### Collision Detection

Instead of maintaining a separate logical map array, the game reads **directly from the live video buffer** at the proposed new position:

```asm
check_collision:
    mov di, dx
    mov ax, [es:di]       ; Read the word at the prospective new cell
    cmp ah, 0x1E          ; Is the attribute byte the blue-wall attribute?
    je game_loop          ; Yes → block the move, restart loop
    cmp di, [finishPos]   ; Is this the goal offset?
    je win_screen         ; Yes → player wins
    ; otherwise, commit the move
```

This technique (treating the screen as the world state) is a classic DOS game trick — the display *is* the game map.

### Win Condition

When `DI` equals `finishPos` (offset `164`, i.e., row 1, column 41), the game jumps to `win_screen`, which clears the display and floods every cell with attribute `0x20` (green background) before exiting.

Copy `build/maze.com` to a bootable DOS disk or USB. Run:

```
maze.com
```

The `.COM` format is fully compatible with MS-DOS 2.0+ and FreeDOS.

## Technical Deep Dive

### INT 16h – Keyboard Input

```asm
mov ah, 0
int 0x16      ; Blocks until a key is pressed
              ; Returns: AH = scancode, AL = ASCII (0 for extended keys)
```

Arrow keys are **extended keys** — they return `AL = 0` and `AH = scancode`. That's why comparisons are done on `AH`:

```asm
cmp ah, 0x48   ; Up arrow
cmp ah, 0x50   ; Down arrow
cmp ah, 0x4B   ; Left arrow
cmp ah, 0x4D   ; Right arrow
```

ESC returns as a normal ASCII key in `AL`:

```asm
cmp al, 27     ; ESC = ASCII 27
```

### INT 21h – Clean Exit

```asm
mov ax, 0x4c00
int 0x21
```

Function `4Ch` (terminate with return code). Using `0x00` as the return code signals success to the parent shell. This is the correct way to exit a DOS program — it flushes file handles, restores interrupt vectors, and returns control cleanly without hanging the system.

### Scancode Reference

| Key | AH Scancode |
|---|---|
| Up Arrow | `0x48` |
| Down Arrow | `0x50` |
| Left Arrow | `0x4B` |
| Right Arrow | `0x4D` |
| ESC | `AL = 0x1B` (ASCII) |

### Video Memory Layout

```
Offset 0     = Row 0, Column 0   (top-left)
Offset 2     = Row 0, Column 1
...
Offset 158   = Row 0, Column 79  (top-right)
Offset 160   = Row 1, Column 0   (start of second row)
...
Offset 3998  = Row 24, Column 79 (bottom-right)
```

Formula: `offset = (row × 80 + col) × 2`

So `finishPos = 164` → `164 / 2 = 82` → row `82 / 80 = 1`, col `82 % 80 = 2`. The goal is at **row 1, column 2**.

`playerPos = 2160` → `2160 / 2 = 1080` → row `1080 / 80 = 13`, col `1080 % 80 = 40`. The player starts at **row 13, column 40** (roughly center-screen).

## Extending the Game

Here are ideas to build on this foundation:

- **Bigger maze**: Replace the flat border-only layout with a full 80×25 maze loaded from a data table or encoded in the `.asm` file.
- **Left/right border walls**: Add column-based wall checks to prevent the player from wrapping around screen edges.
- **Score/timer**: Use BIOS `INT 1Ah` to read the system clock and display elapsed ticks.
- **Multiple levels**: After a win, reset positions and draw a new maze layout.
- **Sound**: Use `INT 10h` or direct port I/O to the PC speaker for movement/win sounds.
- **Enemy movement**: Add a simple enemy tile that moves toward the player each turn.

---

## Known Limitations

- No left/right boundary walls — the player can walk off the edge of a row and appear at the start of the next row (video buffer wrap-around).
- No input debouncing — key repeats from held keys will fire multiple moves.
